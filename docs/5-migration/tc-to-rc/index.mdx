# translationCore to Resource Container Migration

This section provides comprehensive guides for converting translationCore (TC) repositories to standard Resource Container (RC) format. translationCore projects focus on single books with rich word alignment data and tool-specific metadata.

## ğŸ¯ **Migration Overview**

### **Why Integrate TC Books into RC?**
- **Multi-Book Consolidation**: Combine single-book TC projects into complete Bible RC
- **Workflow Continuity**: Maintain translation progress across books in unified repository
- **Content Preservation**: Preserve all translation work and alignment data from TC
- **Team Collaboration**: Enable multiple translators to work on different books
- **Format Standardization**: Convert from tool-specific to ecosystem standard format

### **translationCore Characteristics**
Based on the [translationCore Format Guide](../../4-repository-formats/tool-generated/translationcore-format-guide):

- **Single Book Focus**: Each repository contains one Bible book
- **Two USFM Files**: Simple and complete versions (app-generated only)
- **Rich Alignment Data**: Extensive word-level alignment markers
- **Tool-Specific Metadata**: translationCore-specific manifest format
- **App Data Directory**: `.apps/translationCore/` with tool data

## ğŸ”„ **Integration Process Overview**

### **Step 1: TC Repository Analysis**
- Identify translationCore format using detection patterns
- Analyze file structure and select primary USFM file
- Extract book ID, language, and resource information

### **Step 2: Target RC Assessment**
- **Existing RC**: Identify target RC repository for integration
- **New RC**: Determine if new RC creation is needed
- Analyze target language/resource combination

### **Step 3: Integration Strategy Selection**
- **Add New Book**: TC book doesn't exist in target RC
- **Replace Existing**: TC book replaces existing book in RC
- **Create New RC**: No suitable RC exists for integration

### **Step 4: Content Integration**
- Select and rename USFM file according to RC conventions
- Preserve all alignment data and USFM structure
- Handle file organization within RC structure

### **Step 5: Manifest Updates**
- **Existing RC**: Update manifest to include new/updated book
- **New RC**: Create complete dublin_core manifest structure
- Map TC dependencies to RC relation format

### **Step 6: Validation and Testing**
- Validate RC structure and book integration
- Test alignment data integrity
- Verify multi-book RC consistency

## âš ï¸ **Migration Considerations**

### **Key Challenges**
- **File Selection**: Choose between initial import and working USFM files (always use working file)
- **Alignment Complexity**: Rich alignment data must be preserved exactly
- **Single Book Scope**: TC's single-book focus vs RC's multi-book capability
- **Tool Dependencies**: TC-specific dependencies may not map directly

### **Best Practices**
- **Preserve All Alignment Data**: Never strip alignment markers
- **Always Use Working Files**: Use `{target_language.id}_{resource.id}_{project.id}_book.usfm` (contains all translation work)
- **Maintain Dependencies**: Map all source relationships accurately
- **Test Thoroughly**: Verify alignment data works in RC tools

### **Data Preservation Priorities**
1. **USFM Content**: All text and markup preserved exactly
2. **Alignment Data**: Word-level alignment markers intact
3. **Metadata**: Project, resource, and language information
4. **Dependencies**: Source text relationships maintained
5. **Tool Data**: Optional but valuable for advanced features
