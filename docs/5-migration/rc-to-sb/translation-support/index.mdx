# Translation Support Resource Container to Scripture Burrito Migration

This section covers migration of translation support repositories from Resource Container to Scripture Burrito format. These resources provide guidance, definitions, and training materials for Bible translators.

## 📝 **Translation Support Migration Guides**

### **[Translation Notes](translation-notes)**
Migration guide for verse-specific translation guidance:
- **TN** (Translation Notes) - Bible verse guidance
- **OBS-TN** (OBS Translation Notes) - Story-specific guidance

**Characteristics**:
- ⭐⭐ **Complexity**: Low-Medium
- 📝 **Content**: TSV files with verse-specific notes
- 🎯 **Target Flavor**: parascripturalWordAlignment
- ⏱️ **Time**: 2-3 hours

### **[Translation Words](translation-words)**
Migration guide for biblical term definitions:
- **TW** (Translation Words) - Biblical term definitions and explanations

**Characteristics**:
- ⭐⭐ **Complexity**: Low-Medium
- 📝 **Content**: Nested Markdown files
- 🎯 **Target Flavor**: glossary
- ⏱️ **Time**: 2-3 hours

### **[Translation Academy](translation-academy)**

- **TA** (Translation Academy) - Translation methodology and training

- ⭐⭐⭐ **Complexity**: Medium
- 📝 **Content**: Multi-file Markdown articles
- 🎯 **Target Flavor**: peripheralResource

**Characteristics**:
- ⭐⭐ **Complexity**: Low-Medium

- 🎯 **Target Flavor**: parascripturalWordAlignment
- ⏱️ **Time**: 2-3 hours

## 🎯 **Why Migrate Translation Support Repositories**

### **Benefits of Scripture Burrito for Translation Support**
- **Enhanced Cross-Referencing**: Better relationship modeling between resources
- **Precise Content Typing**: Explicit MIME types for TSV and Markdown content
- **Improved Organization**: Better handling of complex file structures
- **Tool Integration**: Access to specialized Scripture Burrito tools
- **Workflow Support**: Better support for translation workflows

### **Migration Advantages**
- **No Data Loss**: All RC content is preservable and enhanced
- **Better Relationships**: Improved dependency and cross-reference modeling
- **Enhanced Metadata**: Richer content description and organization
- **Future-Proof**: Industry standard format for translation tools

### **Relationship Modeling**
- **Resource Dependencies**: Map RC relations to SB relationships
- **Internal Links**: Preserve cross-references within resources
- **External Links**: Maintain links to other resources (Bible texts, etc.)

## 📋 **Migration Process Overview**

All translation support migrations follow similar steps:

### **Step 1: Analyze Source Structure**
- Examine file organization and content types
- Identify cross-references and dependencies
- Map content to appropriate SB flavor

### **Step 2: Create SB Metadata**
- Convert dublin_core to identification
- Set appropriate flavor type
- Map relationships and dependencies

### **Step 3: Process Content**
- Convert files to appropriate ingredients
- Set correct mimeType and role
- Create precise scope objects
- Preserve cross-references

### **Step 4: Enhance with SB Features**
- Add semantic categorization
- Improve relationship modeling
- Include workflow-specific metadata

### **Step 5: Validate and Test**
- Check SB metadata structure
- Verify content integrity
- Test cross-references
- Validate with SB tools
